## 1. 项目概述

AI Hubs 是一个**新一代智能 Agent 平台**，目标是提供一套"流通且易用"的 AI 框架，解决现有框架配置繁琐、技能下载难、RAG 内存过大、Agent 配置复杂、工作流不全面等问题。

其核心能力包括：

- **快速 API/模型配置**：提供"快速 + 详细"双模式配置体验。
- **GitHub 技能市场集成**：内置技能 + 一键从 GitHub 导入开源技能。
- **多层记忆保障防幻觉**：Git 式 commit 回退、记忆图谱索引、高无损 LLM 压缩。
- **多端支持**：CLI + Web + Electron 桌面端 + 后台管理。
- **全功能管理**：用户注册登录、Agent / 任务 / 对话 / 技能 / 数据集管理。
- **内置 IDE**：直接在浏览器中编写、运行、调试代码。
- **UI 简洁不偷懒**：功能完整但界面克制。

### 1.1 设计原则

1. 质量最优先，测试全面不遗漏。
2. 每次改动前提交 Git，保证可追溯。
3. 默认工作于自主闭环，减少用户等待确认。
4. 实时数据优先使用工具获取，不依赖训练数据凭空作答。

## 2. 功能特性

### 2.1 多 Agent 编排（8 种模式）

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| `single` | 单 Agent 执行 | 简单问答、单一操作 |
| `parallel` | 多 Agent 并行 + 汇总 | 多角度分析、对比研究 |
| `pipeline` | 串行流水线接力 | 多步骤任务、数据处理 |
| `collaborative` | 协作讨论互审 | 决策评估、方案评审 |
| `debate` ⭐ | 正反方辩论 + 投票裁决 | 技术选型、利弊分析 |
| `peer_review` ⭐ | 执行→评审→修改→确认 | 代码审查、质量保障 |
| `round_table` ⭐ | 圆桌会议 + 共识追踪 | 头脑风暴、团队决策 |
| `hierarchical` ⭐ | 专家→经理→总监层级 | 审批流程、重大决策 |

### 2.2 邮箱验证码注册

- QQ 邮箱 SMTP 验证码发送（HTML 邮件）。
- 密码强度校验 + 确认密码，客户端预校验 + 60 秒冷却倒计时。
- JWT Token 认证（包含 `uid` / `role`）。

### 2.3 技能市场

- **8 个内置技能**：Python 开发、Web 全栈、数据分析、文档写作、代码审查、DevOps、调研分析、UI 设计。
- **GitHub 技能探索**：搜索开源 AI 技能仓库，一键导入安装。
- **自定义创建**：定义提示词模板、分类、标签。
- **完整 CRUD**：安装 / 卸载 / 删除 / 创建。

### 2.4 增强记忆系统（防幻觉核心）

- **Git 式 VCS**：`commit` / `checkout` / `log` / `diff` 记忆版本控制。
- **记忆图谱索引**：关键词提取、语义关联边、主题聚类（BFS 图谱查询）。
- **高无损 LLM 压缩**：替代简单截断，信息损失 < 5%。
- **双路检索**：图谱关键词 + 向量语义检索。

### 2.5 内置 IDE

- 多文件标签管理（新建 / 保存 / 关闭）。
- 语法高亮（Python / JS / TS / HTML / CSS / JSON / MD / SQL / YAML）。
- 行号 + 文件浏览器 + 底部状态栏，Ctrl+S 保存。

### 2.6 对话即管理

- 斜杠命令：`/agent` `/task` `/clear` `/recall` `/code` `/skill` `/search` `/config`
- 快捷操作面板：一键跳转 Agent / 任务 / 技能 / IDE。
- SSE 实时流式对话 + Markdown 渲染。

### 2.7 数据集管理

- CRUD + JSON / CSV / 纯文本导入导出。
- 文件上传 + 追加记录。
- 分类：训练数据 / 测试数据 / 提示词模板 / 知识问答 / 自定义。

### 2.8 多端支持

- **Web 端**：React 18 + TypeScript + Vite 6。
- **CLI 端**：`prompt_toolkit` + Tab 补全 + 历史回溯 + Rich 渲染。
- **桌面端**：Electron 28（Win / Mac / Linux）。

## 5. 核心模块详解

### 5.1 核心引擎（`src/core`）

- **`agent.py`**：基于 LangGraph 实现 ReAct 循环（思考 → 调用工具 → 观察 → 再思考），`max_iterations` 控制单次任务最大迭代。
- **`llm.py`**：统一 LLM 抽象，支持 5 种提供商；`api_key` 留空时按 provider 从环境变量读取（`DEEPSEEK_API_KEY` 等）。
- **`orchestrator.py`**：实现 8 种编排模式（见 2.1），`default_mode: auto` 时根据任务描述自动选择（字数复杂度、并行 / 流水线关键词评分）。
- **`task_manager.py`**：任务队列，支持任务编排与状态跟踪。
- **`config.py`**：加载并校验 `config.yaml`，提供运行时配置访问。

### 5.2 记忆系统（`src/memory`）

- **`memory_manager.py`**：短期记忆（最近 N 轮，`summarize_threshold` 触发自动摘要）+ 长期记忆（ChromaDB 持久化）。
- **`enhanced_memory.py`**：
  - **VCS**：对话过程自动 commit，支持 `log` / `checkout` / `diff`，可回退到任意版本（防幻觉的关键保障）。
  - **记忆图谱**：关键词提取 + 语义关联边 + 主题聚类，BFS 查询关联记忆链。
  - **LLM 压缩**：调用 LLM 智能压缩对话历史，压缩率 5–10x，信息损失 < 5%。

### 5.3 技能市场（`src/skills`）

- **`skill_manager.py`**：8 个内置技能定义 + 自定义技能 CRUD + 安装 / 卸载。
- **`github_scanner.py`**：调用 GitHub 搜索 API 发现开源技能仓库并一键导入。

### 5.4 工具生态（`src/tools`）

17 个内置工具（网络搜索、网页抓取、文件读写、Python 执行、计算器、图像生成、数据库查询、HTTP API 调用、PDF / Excel 读取、JSON 处理、时间、文本 diff、图像分析等）。`config.yaml` 的 `tools.enabled` 列表决定启用哪些，`tools.dangerous` 列表标记需确认的危险工具。`max_calls_per_turn` 限制单轮调用次数。

### 5.5 RAG 知识库（`src/rag`）

- 文档切分（`chunk_size` / `chunk_overlap`）→ 嵌入 → 存入 ChromaDB。
- 检索结果支持压缩（`summary` / `extractive` / `hybrid`）以减少上下文占用，`top_k` 控制返回条数。

### 5.6 认证（`src/auth`）

- JWT 认证，Token 含 `uid` / `role`。
- 支持邮箱 + SMTP 验证码注册、登录、用户信息获取。
- `bcrypt_rounds` 控制密码哈希强度。

### 5.7 界面层（`src/ui`）

- **`web_server.py`**：FastAPI 主服务，注册 12 个路由模块（见第 8 章），提供 REST + SSE + WebSocket。
- **`cli.py`**：CLI 交互，支持 Tab 补全、历史回溯、`/help` 查看命令。
- **`routers/`**：按模块拆分路由，便于维护与权限控制。

## 6. 配置说明（`config.yaml`）

> 生产环境建议将敏感字段（数据库密码、JWT 密钥、LLM Key）通过 `.env` 或环境变量注入，**不要明文写入 `config.yaml`**。

| 配置段 | 关键字段 | 说明 |
|--------|----------|------|
| `llm` | `provider` / `model` / `api_key` / `temperature` / `max_tokens` | 主模型配置；`api_key` 留空则从环境变量读取 |
| `fallback` | `provider` / `model` | 主模型失败时的备用模型 |
| `memory` | `short_term.max_turns` / `long_term.enabled` / `strength.*` | 短期轮数、长期开关、记忆强度与衰减 |
| `rag` | `embedding_model` / `top_k` / `compression_*` | 知识库切分、检索、压缩策略 |
| `tools` | `enabled` / `dangerous` / `max_calls_per_turn` / `output_dir` | 工具启用、危险标记、单轮上限、文件根目录 |
| `agent` | `name` / `max_iterations` / `system_prompt` | Agent 身份、迭代上限、系统提示词 |
| `server` | `host` / `port` | Web 服务监听（默认 `127.0.0.1:8080`） |
| `database` | `host` / `port` / `user` / `password` / `database` | MySQL 连接；**留空则内存模式** |
| `auth` | `enabled` / `jwt_secret_key` / `jwt_expire_minutes` | 认证开关、JWT 密钥（生产必填）、有效期 |
| `rate_limit` | `enabled` / `max_requests_per_minute` / `burst` | 每 IP 速率限制 |
| `logging` | `level` / `dir` / `json_format` / `mysql_errors` | 日志级别、目录、JSON 格式、错误入库 |
| `orchestrator` | `default_mode` / `max_agents` / 各模式参数 | 编排模式与阈值 |
| `monitoring` | `enabled` / `metrics_path` / `health_path` | Prometheus 指标与健康检查路径 |

## 7. 数据存储

### 7.1 MySQL（主持久化）

- 当 `database` 配置完整且可达时，应用自动连接并**建表**，用户 / 任务 / Agent / 对话等结构化数据持久化。
- 表结构由 `src/infrastructure/database.py` 在启动时自动创建。
- 若未配置或不可达，应用**降级为内存模式**（重启即丢数据）。

### 7.2 ChromaDB（向量）

- 记忆长期存储与 RAG 检索，持久化目录见 `memory.long_term.db_path` 与 `rag.persist_dir`。

### 7.3 文件系统

- 工具读写限制在 `tools.output_dir`（部署建议为绝对路径，如 `/data/ai_hubs/output`），不得越权访问系统文件。

## 9. 认证与权限

- 默认 `auth.enabled: true`，所有受保护 API 需携带 `Authorization: Bearer <JWT>`。
- 首次启动若数据库为空，会 **seed 默认管理员账号** `admin` / `admin123`（可通过 `ADMIN_PASSWORD` 环境变量修改，建议登录后立即修改）。
- 速率限制按 IP 生效，可在 `rate_limit` 中配置白名单。

## 10. 部署指南

### 10.1 架构概览（生产推荐）

```
用户浏览器
    │  HTTP 80
    ▼
Nginx（8.138.24.27）
    ├── /            → 静态托管 React SPA（/var/www/ai_hubs/dist）
```

### 10.2 服务器初始化

```bash

apt-get update
apt-get install -y python3.10-venv nginx mysql-server

# 2. 克隆仓库
git clone https://github.com/TYYHeck/ai_hubs.git /root/ai_hubs
cd /root/ai_hubs

# 3. 创建 venv（--system-site-packages 复用已装包）
python3 -m venv --system-site-packages venv
source venv/bin/activate
pip install -r requirements.txt
```

### 10.3 配置 MySQL

```sql
CREATE DATABASE IF NOT EXISTS ai_hubs
  CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER IF NOT EXISTS 'ai_hubs'@'localhost' IDENTIFIED BY 'AiHubs@2024!';
GRANT ALL PRIVILEGES ON ai_hubs.* TO 'ai_hubs'@'localhost';
FLUSH PRIVILEGES;
```

并在 `/root/ai_hubs/.env` 中填入 `DATABASE_*` 与 `JWT_SECRET_KEY`、`ADMIN_PASSWORD`、LLM Key 等（参见第 6 章）。

### 10.4 前端构建与上传

前端构建在**本地**完成（避免服务器资源受限导致 `npm build` 失败）：

```powershell
# 本地（Windows）
cd g:\Documents\Agents_Projetcts\ai_hubs\frontend
npm install
npm run build

# 上传产物到服务器静态目录
ssh root@8.138.24.27 "mkdir -p /var/www/ai_hubs"
```

> **关键坑位**：静态文件必须放在 `/var/www/ai_hubs/dist`，**不能**放在 `/root/ai_hubs/frontend/dist`，因为 `/root` 目录权限为 `700`，Nginx 的 `www-data` 用户无法进入，会导致 500 错误。

### 10.5 systemd 服务

```bash
cp deploy/ai_hubs.service /etc/systemd/system/
systemctl daemon-reload
systemctl enable ai_hubs
systemctl start ai_hubs
```

服务要点（`deploy/ai_hubs.service`）：
- `WorkingDirectory=/root/ai_hubs`，`User=root`
- 通过 `EnvironmentFile=/root/ai_hubs/.env` 注入环境变量
- 启动命令：`venv/bin/python main.py --web --host 0.0.0.0 --port 8080`
- `Restart=always`，内存上限 `MemoryMax=4G`

### 10.6 Nginx 反代

```bash
cp deploy/nginx.conf /etc/nginx/sites-available/ai_hubs
ln -s /etc/nginx/sites-available/ai_hubs /etc/nginx/sites-enabled/
nginx -t
systemctl reload nginx
```

Nginx 行为：
- `location /`：托管 React SPA，`try_files $uri /index.html;`（**务必去掉 `$uri/`**，否则根路径会触发内部重定向死循环 → 500）。
- `location /assets/`：7 天强缓存。
- `/api/`、`/ws`、`/health`、`/metrics`：反代到 `127.0.0.1:8080`。
- `/api/chat`：关闭 buffering 以支持 SSE 流式。
- `/metrics`：仅限内网 IP。

### 10.7 端口与防火墙

- 后端监听 `8080`，仅绑定内网；对外只暴露 Nginx 的 `80`。
- 确保 `8080` 端口**不被其他进程占用**（详见第 12 章排查）。

## 12. 常见问题排查

### 12.1 端口 8080 被占用，服务起不来

```bash
ss -ltnp | grep 8080          # 查看占用进程
ps -eo pid,cmd | grep main.py # 确认是否为旧项目
kill <pid>                    # 停止旧进程
systemctl reset-failed ai_hubs
systemctl start ai_hubs
```

> 历史案例：旧项目 `/root/smart_agent` 曾持续占用 8080，需 `systemctl disable smart_agent && rm -rf /root/smart_agent` 彻底清理。

### 12.2 Nginx 返回 500（静态文件读不到）

原因：静态根目录设在 `/root/...` 下，Nginx 无权限。解决：将 `dist` 放到 `/var/www/ai_hubs/dist`，并同步修改 `/etc/nginx/sites-available/ai_hubs` 的 `root`。

### 12.3 根路径 `/` 返回 500，但 `/index.html` 正常

原因：`try_files $uri $uri/ /index.html;` 在根路径触发重定向死循环。解决：改为 `try_files $uri /index.html;`。

### 12.4 对话无响应 / 模型不可用

`config.yaml` 或 `.env` 中 LLM `api_key` 为占位符时，对话无法调用模型。需在**系统设置 → 模型 / LLM 配置**填入真实 Key（如 DeepSeek），或设置 `DEEPSEEK_API_KEY` 等环境变量。

### 12.5 数据库连不上，数据丢失

检查 `config.yaml` 的 `database` 段与 `.env` 的 `DATABASE_*` 是否一致，MySQL 是否运行，用户权限是否到位。未配置数据库时应用会降级为内存模式。

### 12.6 推送 GitHub 失败（workflow 权限）

若提交中包含 `.github/workflows/*.yml` 且 Token 无 `workflow` 权限，推送会被拒。本项目采用 SSH 手动部署，可移除该 Workflow 文件。

## 15. 客户端（Electron 桌面端）

AI Hubs 提供 Electron 桌面客户端（Win / Mac / Linux），将前端 SPA 与后端 Python 进程打包进同一个桌面应用，启动后自动拉起后端并通过内置浏览器窗口加载界面。

### 15.1 相关文件

| 文件 | 作用 |
|------|------|
| `package.json`（根目录，name=`ai-hubs-desktop`） | 客户端入口与打包配置，`main: electron/main.js` |
| `electron/main.js` | Electron 主进程：启动后端子进程、创建窗口、自定义菜单、生命周期管理 |
| `electron/preload.js` | 预加载脚本：以 `contextBridge` 向渲染进程安全暴露窗口控制 / 文件对话框 / 通知 API |
| `frontend/dist/` | 打包时纳入的已构建前端静态资源 |

### 15.2 构建与运行命令

```bash
# 开发模式（需先 npm install 安装 electron / electron-builder）
npm install
npm run frontend:build     # 先构建前端
npm run electron:dev       # 启动 Electron（electron .）

# 打包（输出到 dist-electron/）
npm run electron:build     # win=nsis, mac=dmg, linux=AppImage
```

打包配置（`package.json` 的 `build` 段）已包含 win / mac / linux 三类目标，产物会打包 `electron/`、`frontend/dist/`、`main.py`、`src/`、`config.yaml`、`requirements.txt`。
